Concept

Away in Berlin is an archive asking, what happens to a memory when it passes from phone to screen, through platforms, and back again? In the summer of 2023, I spent three months studying abroad in Berlin, an important and lively period of my life that I documented obsessively and chaotically through photography, video, and social media. But when I went to revisit those memories, I found something unexpected — many of the images I thought I had saved existed only as degraded Instagram stories or random screenshots, their quality eroded by the very platform I used to share them.

This project is an attempt to remember, collect, organize, and present that archive as it actually exists — in its full, sometimes broken state. Low-resolution screenshots sit alongside high-quality camera photos. Missing files appear as gray boxes. The decay, just like memory, is the archive.

Central Question What does it mean to archive digital memories when the platforms and devices that hosted them don't allow you to retrieve them intact?
Process
Wk 1–6
Lectures and discussions on digital decay and preservation, grounding the course in the understanding that born-digital works erode as their sustaining systems deteriorate. Archival case studies, metadata strategies, and hands-on HTML/CSS exercises introduced frameworks for documenting absence and loss, alongside readings on the ethics of exhibiting incomplete works.
Week 7
Defined the concept and scope. Began wireframing in Figma. Explored and ideated the archive's visual language — monospace typography, cyan sidebar, grid-based layout — drawing from museum catalog and database aesthetics.
Week 8
Built the initial website, using Claude's assistance for the first time to build HTML, CSS, and JavaScript. Implemented the photo grid, sidebar filters, detail view overlay, and search functionality. Connected the site to GitHub for live hosting via GitHub Pages.
Week 9
Curated a selection of 1,152 images and videos from the original collection of 1,700+. Wrote a Python script to automatically scan the images folder and generate a spreadsheet (CSV) with all file metadata. Began manually tagging each item with title, category, vibe, and descriptive tags in Google Sheets.
Week 10
Migrated all media to Cloudinary for cloud-based hosting. Updated the website data to reflect real Cloudinary URLs. Resolved multiple Git and file-size conflicts during the push to GitHub. Got the site live with real photos, working filters, and a functional archive.

Technical Stack
HTML / CSS / JS
Core website built without frameworks. Chosen for simplicity, full control, and learning value.
GitHub Pages
Free static site hosting connected directly to the VS Code project via Git version control.
Cloudinary
Cloud storage and delivery for 1,152 media files. Enables fast image loading without bloating the GitHub repository.
Python
Used to write automation scripts: scanning the images folder, generating CSVs, and bulk-uploading to Cloudinary via Terminal.
Google Sheets
Used as a content management system for tagging and categorizing all 1,152 archive items with titles, categories, vibes, and tags.
Claude (Anthropic)
AI assistant used throughout development for writing code, debugging, and learning concepts in plain English.
Challenges
  • GitHub file size limits. Attempting to push 1,152 images to GitHub repeatedly failed with HTTP 500 errors. The solution was to migrate all media to Cloudinary and use a .gitignore file to exclude the images folder — keeping GitHub only for code.
  • Cloudinary browser upload failures. Uploading 1,152 files through the browser interface repeatedly timed out and failed after hours of attempts. The final solution was a Python script run through Terminal, which completed the upload reliably without browser timeouts.
  • Cloudinary public ID renaming. On the first upload attempt, Cloudinary appended random suffixes to all filenames, breaking the URL mapping. This required deleting all assets, adjusting the upload preset settings, and re-uploading with the original filenames preserved.
  • Filter logic complexity. Building a filter system that correctly handled both file type (images/video/screenshot) and subject category (food, art, landscape) required multiple iterations to separate the two filter sets and define the correct logic for each.
  • Git history conflicts. After accidentally committing the images folder to Git, every push attempt tried to send 3+ GB of data. The solution was to create a fresh Git branch with no history (--orphan), commit only the HTML file, and force push to the new repository.
Reflection

Building this archive was itself a lesson on digital decay. Files went missing, uploads failed, and platforms imposed their own logic on my images. The 725 photos that couldn't be uploaded in time by class in week 8, the gray boxes where images should be, the Instagram screenshots that exist only because the platform wouldn't let me download the originals — none of this was planned, most of it was incredibly frustrating, but all of it became part of the work.

What surprised me most was how much the technical process mirrored the conceptual one. Every challenge — a Git conflict, a timed-out upload, a renamed file — was a process of deterioration and distortion.

I feel very rewarded as I have never coded a complete website before. I didn't do it entirely on my own, but learning about code through the development of an archive has been challenging. Still, it gave me a boost of confidence to use coding as a tool in my creative process going forward. The archive is not complete. It may never be. Away in Berlin was a moment in time.
